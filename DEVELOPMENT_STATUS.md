# safeDoc 개발 상태 및 재개 지점

작성 기준: 2026-07-12 현재 작업 디렉터리

## 현재 상태

- TXT, CSV, XLSX, DOCX, HWPX 파서와 개인정보 탐지·검토·비식별 처리 기능이 구현되어 있다.
- 단위시험 52개는 모두 통과한다.
- 단일 HTML 프로덕션 빌드는 성공한다.
- PDF 처리의 UI 비동기 연결과 기본 단위시험을 완료했다.
- Git 저장소가 초기화되어 있지 않아 변경 이력이나 마지막 정상 커밋을 기준으로 비교할 수 없다.

## 완료한 복구 작업: PDF 연결

PDF 관련 라이브러리, `src/parsers/pdf.js`, 서브셋 글꼴, `src/core/analyzer.js` 분기를 UI에 연결했다.

1. `src/core/fileManager.js`
   - `pdf`를 `SUPPORTED_EXTENSIONS`로 이동했다.
2. `src/main.js` 분석 흐름
   - `parseDocument()`를 `await`하도록 수정했다.
3. `src/main.js` 결과 생성
   - `executeDeidentify()`를 비동기로 변경하고 `buildResult()`를 `await`하도록 수정했다.
4. `src/main.js` 복원 흐름
   - `parseDocument()`와 `restoreDocument()`를 모두 `await`하도록 수정했다.
5. `index.html`
   - 텍스트 PDF 지원 및 PDF 결과의 검색·복사·복원 제한을 안내한다.
6. 시험
   - `test/pdf.test.js`에 텍스트 추출, 공통 비동기 진입점, 결과 생성, 원문 텍스트 제거, 스캔 PDF 거부, 복원 거부 시험을 추가했다.
   - 암호 PDF, 페이지 제한 및 실제 브라우저 렌더링을 검증하는 통합시험은 추가로 필요하다.

## 요구사항과 충돌하여 결정이 필요한 사항

현재 요구사항 정의서와 기술명세서는 복원 기능을 제공하지 않고 대응표를 별도 파일로 저장하지 않도록 명시한다(FR-510, FR-703 및 기술명세서 VII-5). 그러나 현재 프로그램에는 다음 기능이 구현되어 있다.

- 대응표 JSON 다운로드
- 대응표를 이용한 TXT, CSV, XLSX, DOCX, HWPX 복원
- 별도의 “원본 복원” 화면

다음 중 하나를 제품 책임자가 결정해야 한다.

- 명세 준수: 대응표 다운로드와 복원 UI·코드를 제거한다.
- 기능 유지: 요구사항 정의서, 기술명세서, 보안 안내와 검수 기준을 먼저 개정한다.

이 결정 전에는 복원 기능을 확장하거나 PDF 복원 UX를 추가하지 않는다.

## 명세 대비 주요 미완성 항목

- FR-104: 분석 화면에서 시트 수·페이지 수 등 파싱 후 메타정보가 실제로 표시되지 않는다.
- FR-203: XLSX 내용을 시트별로 구분하여 표시하지 않는다.
- FR-206/NFR-104: 분석 취소는 단계 사이에서만 확인하며 파싱·탐지 중에는 동작하지 않는다. Web Worker 이전이 필요하다.
- FR-314: 일반적인 필드명 주변 값 탐지는 성명·주소 및 CSV 헤더 일부에 한정된다.
- FR-406: 동일 값 일괄처리는 있으나 동일 유형 전체 일괄선택은 없다.
- FR-802: 사용자 지정 치환문구 설정이 없다.
- NFR-103: 20MB 입력을 허용하지만 ZIP은 압축 해제 후 최대 300MB까지 메모리에 유지하며 PDF는 페이지 이미지를 추가 보관한다. 실제 한도 시험이 없다.
- 탐지 정확도 목표: 승인된 블라인드 데이터셋과 정밀도·재현율 측정 결과가 없다.
- 산출물: 요구사항 추적표, 설계서, 규칙 설명서, 통합·정확도·보안 시험 결과서, 라이선스 목록, 사용자 및 유지보수 매뉴얼이 없다.

## 확인된 기술 위험

- 사용자 정규표현식은 메인 스레드에서 실행되어 복잡한 패턴이 화면을 멈출 수 있다.
- 복원 토큰이 문서에 원래 존재하던 문자열과 충돌할 수 있다.
- 대응표 파서는 배열 존재 여부 외에 항목 타입·크기·버전을 검증하지 않는다.
- ZIP 안전 한도는 `unzipSync()` 실행 후 검사되므로 압축 폭탄에 대한 사전 방어가 아니다.
- Office/HWPX XML 처리는 지정된 텍스트 노드만 대상으로 하므로 도형, 일부 개체, 문서 속성 등에 개인정보가 남을 수 있다.
- PDF 페이지 전체를 2배율 PNG로 메모리에 보관하므로 긴 문서에서 메모리 사용량이 매우 커질 수 있다.
- PDF 텍스트 좌표 계산은 글자별 실제 glyph 폭이 아니라 문자열 길이 비례 계산이어서 가림 영역이 부정확할 수 있다.

## 권장 재개 순서

1. 복원·대응표 기능을 유지할지 명세대로 제거할지 결정한다.
2. PDF 비동기 연결부를 수정하고 PDF 단위시험·UI 통합시험을 추가한다.
3. 실제 브라우저에서 단일 HTML을 오프라인 실행하여 PDF Worker와 글꼴 포함 여부를 검증한다.
4. 분석 작업을 Web Worker로 옮기고 취소·진행률을 실제 작업 단위로 구현한다.
5. 실제 Office/HWPX/PDF 샘플을 이용한 결과 파일 호환성 시험을 추가한다.
6. 요구사항 추적표를 작성하고 필수 요구사항 누락부터 닫는다.
7. 정확도·성능·보안 시험과 제출 문서를 완성한다.

## 현재 검증 명령

PowerShell 실행 정책 때문에 `npm` 대신 `npm.cmd`를 사용한다.

```powershell
npm.cmd test
npm.cmd run build
```

현재 결과: 시험 52개 통과, 빌드 성공. PDF의 실제 브라우저 캔버스 렌더링과 다운로드 결과 육안 검증은 남아 있다.
